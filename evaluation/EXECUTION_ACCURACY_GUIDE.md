## 🎯 **What is Execution Accuracy (EX)?**

**Execution Accuracy (EX)** is the most reliable Text-to-SQL evaluation metric because it measures **semantic correctness** rather than syntactic similarity.

### How it Works:

1. **Execute Ground Truth SQL** against the database
2. **Execute Predicted SQL** (generated by agent) against the same database
3. **Compare Result Sets** (not the SQL text itself)
4. **Score**: 1.0 if results match exactly, 0.0 otherwise

### Why EX is Critical:

```sql
Ground Truth: SELECT COUNT(*) AS total_internacoes FROM internacoes;
Predicted:    SELECT COUNT(*) FROM internacoes;

Exact Match (EM):     0.0 ❌  (different structure - missing alias)
Component Match (CM): 0.925 ✅ (structure mostly correct)
Execution Accuracy (EX): 1.0 ✅  (SAME RESULT when executed!)
```

**Key Insight**: The predicted SQL is **semantically correct** even though it's syntactically different. Only EX captures this!

## 🔧 **Implementation Details**

### Database Connection

The EX metric requires a live database connection:

```python
from evaluation.metrics.execution_accuracy import ExecutionAccuracyMetric

# Initialize metric with timeout
ex_metric = ExecutionAccuracyMetric(execution_timeout=30)

# Evaluate with database connection
context = EvaluationContext(
    question_id="GT001",
    question="Quantas internações?",
    ground_truth_sql="SELECT COUNT(*) AS total FROM internacoes;",
    predicted_sql="SELECT COUNT(*) FROM internacoes;",
    database_connection=db_connection  # ← Required!
)

result = ex_metric.evaluate(context)
# result.score = 1.0 (results match)
```

### Result Comparison Algorithm

The EX metric uses sophisticated comparison:

1. **Execute both queries** with timeout protection
2. **Normalize results**:
   - Convert types (Decimal → float, etc.)
   - Handle NULL values
   - Normalize whitespace in strings
3. **Compare using Counter** (multiset comparison):
   - Handles row order differences
   - Detects missing/extra rows
   - Provides detailed diff on mismatch

```python
# Example normalization
gt_results = [(Decimal('11000000'),)]
pred_results = [(11000000,)]

# After normalization:
gt_normalized = [(11000000.0,)]
pred_normalized = [(11000000.0,)]

# Counter comparison
Counter(gt_normalized) == Counter(pred_normalized)  # True ✅
```

### Error Handling

The metric handles various failure modes:

```python
# Case 1: Ground truth fails to execute
{
  "score": 0.0,
  "is_correct": False,
  "details": {
    "reason": "Ground truth query execution failed",
    "ground_truth_error": "column 'xyz' does not exist"
  }
}

# Case 2: Predicted query fails
{
  "score": 0.0,
  "is_correct": False,
  "details": {
    "reason": "Predicted query execution failed",
    "predicted_error": "syntax error at or near 'FROM'",
    "ground_truth_executed": True,
    "ground_truth_rows": 1
  }
}

# Case 3: Results mismatch
{
  "score": 0.0,
  "is_correct": False,
  "details": {
    "results_match": False,
    "comparison_details": {
      "size_mismatch": True,
      "gt_size": 10,
      "pred_size": 15,
      "size_difference": 5
    }
  }
}
```

## 📈 **Interpreting EX Results**

### Score Interpretation:

| EX Score | Meaning | Action |
|----------|---------|--------|
| **1.0** | Perfect semantic match | Query is correct ✅ |
| **0.0** | Results don't match OR execution failed | Query is wrong ❌ |

### Common Scenarios:

#### Scenario 1: High EX, Low EM
```
EM: 0.0, CM: 0.85, EX: 1.0
```
**Interpretation**: Query is semantically correct but syntactically different.
**Example**: Missing alias, different column order, equivalent WHERE clauses.
**Verdict**: ✅ Acceptable for production

#### Scenario 2: High CM, Low EX
```
EM: 0.0, CM: 0.90, EX: 0.0
```
**Interpretation**: Structure looks good but results are wrong.
**Example**: Wrong table, wrong filter values, off-by-one errors.
**Verdict**: ❌ Needs fixing

#### Scenario 3: All High
```
EM: 1.0, CM: 1.0, EX: 1.0
```
**Interpretation**: Perfect match in structure AND semantics.
**Verdict**: ✅ Ideal

#### Scenario 4: All Low
```
EM: 0.0, CM: 0.3, EX: 0.0
```
**Interpretation**: Query is fundamentally wrong.
**Verdict**: ❌ Major issue

## 🔍 **Advanced Features**

### 1. Result Set Normalization

Handles type differences gracefully:

```python
# Ground truth returns Decimal
gt_result = [(Decimal('569123'),)]

# Predicted returns int
pred_result = [(569123,)]

# EX normalizes both to float
# Score: 1.0 ✅
```

### 2. Row Order Insensitivity

Uses Counter for multiset comparison:

```python
# Ground truth
gt = [('A', 1), ('B', 2), ('C', 3)]

# Predicted (different order)
pred = [('C', 3), ('A', 1), ('B', 2)]

# EX uses Counter comparison
Counter(gt) == Counter(pred)  # True ✅
# Score: 1.0 ✅
```

### 3. Detailed Mismatch Analysis

When results don't match, provides diff:

```python
{
  "results_match": False,
  "comparison_details": {
    "missing_rows_count": 2,
    "extra_rows_count": 1,
    "missing_rows_sample": [('John', 25), ('Jane', 30)],
    "extra_rows_sample": [('Bob', 35)]
  }
}
```

### 4. Timeout Protection

Prevents hanging on expensive queries:

```python
ex_metric = ExecutionAccuracyMetric(execution_timeout=30)  # 30 seconds

# Sets PostgreSQL statement_timeout
cursor.execute("SET statement_timeout = 30000")  # 30000ms
```
